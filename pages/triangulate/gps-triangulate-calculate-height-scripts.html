gps triangulate calculate height python script

This page details a custom GIS workflow using Python to triangulate geographic coordinates to estimate height of distant landmarks using EXIF metadata extracted from smartphone photos.

Passing photo coordinates and stable bearing data through a triangulation script (gps-triangulate-pics.py), we can map distant targets (buildings, cell towers, mountain tops, etc) with ~5% relative error margin under real-world conditions. Feeding outputs to a secondary script (gps-calculate-height.py) using camera pitch we calculate target height estimations.

Python script gps-triangulate-pics.py outputs target distance, coordinates, and an ASCII ray graph - to stderr for screen display, while returning clean, pipable geocoords (e.g., -32.225535,115.802791) directly to stdout.

When passed the -a flag, the script formats its stdout as named arguments (e.g., --olat -32.225534732096165 --olon 115.80279056457381) to be piped directly into gps-calculate-height.py.

1. Error Dynamics & Geometric Stability

The system achieves a ~5% relative distance error and stays within ~4.9 metres of the Google Earth control point over an 80 - 100m range. In computational geometry and surveying, this is an incredibly tight result for uncalibrated hardware.

The stability of your calculation relies on three main geometric factors:

  • The Baseline-to-Distance Ratio: In Example 2, your photo capture points are separated by “several hundred metres” while tracking a target ~90m away. This creates a wide baseline with high interior angles between intersecting bearing lines. Wide angles radically minimize the Geometric Dilution of Precision (GDOP). If you had taken the photos only 5 metres apart, the rays would be nearly parallel, causing minor compass errors to displace the intersection point (the “X”) by tens of metres.
  • Least-Squares Smoothing: The ChatGPT excerpt notes that the system converges via a stable least-squares solution. By using three photos instead of the minimum two, your script successfully averages out random high-frequency noise (e.g., local magnetic interference on a single frame) rather than amplifying it.

2. The Sensor Bottleneck: Stock vs. OpenCamera

Your observation regarding the Pixel 7 Pro’s stock camera flipping the GPSImgDirection tag by 180° in portrait mode points to an unresolved Android coordinate mapping bug.

The Android sensor framework maps hardware orientation relative to the device’s natural orientation (typically portrait for phones). When an app requests camera frames and orientation metadata, it must explicitly transform the coordinate space based on screen rotation:

θEXIF=(θSensor+θUI_Rotation)(mod360)\theta_{\text{EXIF}} = (\theta_{\text{Sensor}} + \theta_{\text{UI\_Rotation}}) \pmod{360}

If an app fails to track the active UI rotation state correctly during a capture event - or treats portrait mode as a baseline without accounting for camera sensor alignment - it writes raw or inverted values. OpenCamera avoids this by querying the Android OrientationEventListener directly and manually mapping the device’s 3D rotation matrix (Yaw, Pitch, Roll) onto the global magnetic/true north reference frame.

3. Height Estimation Mechanics & Math

Your secondary script gps-calculate-height.py uses basic right-angle trigonometry to calculate height (HH) based on the triangulated horizontal distance (DD) and camera pitch (α\alpha):

H=D⋅tan(α)H = D \cdot \tan(\alpha)

The Missing Variable: Delta Altitude (ΔA\Delta A)

As noted in your markdown, this assumes a flat baseline (HPhotographer=HTarget BaseH_{\text{Photographer}} = H_{\text{Target Base}}). In reality, the true height of the tower from its own base is bound by the terrain profile:

HTrue=(D⋅tan(α))+(APhotographer−ATarget Base)H_{\text{True}} = (D \cdot \tan(\alpha)) + (A_{\text{Photographer}} - A_{\text{Target Base}})

Because your current script cannot calculate the terrain altitude at the target’s newly triangulated coordinate, your final calculation (15.60 m15.60\text{ m}) measures the height of the tower relative to your eye level at the time of the shoot, not relative to the building’s foundation.

4. Recommendations for Script Upgrades

To take this workflow to an institutional grade, consider implementing the following code logic updates:

  1. Incorporate Digital Elevation Models (DEM): Instead of relying purely on the photo’s internal GPSAltitude (which can have vertical errors up to 3x worse than horizontal GPS data), you can update the script to query a free spatial elevation API (like Open-Elevation or USGS) using the calculated target coordinates to determine true ΔA\Delta A.
  2. Weight the Triangulation Rays: Not all photos are equal. Your script could weight the intersection calculations based on the proximity of the shot or the stability of the phone’s hardware compass flag at the moment of capture, favoring tighter geometric intersections.
